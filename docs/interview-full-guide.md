> 每条按：**可能问法 → 精简回答 → 追问预判 → 指标细节 → 设计思路 → 瓶颈 → 换场景方案** 七维度展开。

## 第 1 条：意图路由 + Skill 插件化

### 可能问法
- "意图路由是怎么做的？"
- "规则判定和 LLM 怎么配合？"
- "Skill 怎么做到零侵入扩展的？"

### 精简回答（30秒）
路由分两层递进：规则层匹配 greeting、预约、取消等高频意图，命中即走不走 LLM；规则判不出的才调轻量模型（light LLM）做语义分流。Skill 通过注册表 + match 接口接入，新增 Skill 只写一个类，路由代码零改动。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "规则和 LLM 的比例多少？" | 规则覆盖率约 60-70%，剩余走 LLM。我们做过消融实验，规则层对响应延迟有约 200ms 的减少 |
| "规则冲突了怎么办？" | 按优先级排序。取消预约优先于预约，Skill 匹配优先于硬编码规则 |
| "'预约前注意什么' 这种有预约词但不是预约意图的怎么办？" | 这是规则层的边界 case。策略是"规则宁可窄一点（高精确率），漏的交给 LLM 兜底（高召回率）"。规则误路由的代价远大于 LLM 多处理一个 query 的代价 |

### 指标细节
- 4 个 Skill：`GreetingSkill`、`MedicalRagSkill`、`MCPSkill`、`AppointmentSkill`
- 规则判定：纯 CPU 计算，毫秒级
- LLM 分流：用 light 模型，结构化输出 `IntentAnalysis`
- route_log 记录了 `decision_source`（rule / llm / skill），可统计覆盖率

### 设计思路
```
用户 query
  → Skill Registry.classify_intent()     ← 第一层：插件化匹配
  → _classify_query_by_rules()          ← 第二层：硬编码规则
  → LLM 结构化输出 IntentAnalysis      ← 第三层：语义兜底
```

### 瓶颈
- 规则硬编码在 Python 里，业务方改不了（应该可配置化）
- 规则和 LLM 之间没有消融实验数据细化各自的贡献
- 复杂组合意图（"先取消协和的预约，再挂仁济的号"）目前不支持

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"用户一句话有多个意图怎么办？（取消协和+挂仁济）"** | 目前不支持复杂组合意图，这是一个已知局限。如果要支持，思路是把 intent_router 从单一分类改为多标签输出，下游串联执行。但这引入了顺序依赖问题——先取消再挂号 vs 先挂号再取消，结果不同 |
| **"Skill 加到 100 个性能会不会崩？"** | 会。当前的 classify_intent() 是串行遍历所有 Skill 的 match()，O(n)。优化方向：给 match() 加快速判出（如关键词前缀索引），或者把 Skill 按 intent 分组，路由时先缩小候选集再匹配 |
| **"如果用户的消息既像 greeting 又像科室问题？"** | 按优先级：Skill > 硬编码规则。GreetingSkill 的 priority 排最高（因为误判 greeting 的代价最小——大不了重新识别），科室匹配排中间，预约排最后。实际上各类意图交集很小，冲突概率不高 |
| **"规则完全覆盖不了的 case 有多少？"** | route_log 里标记了 decision_source，规则覆盖约 60-70%，剩余 30-40% 走 LLM。这 30-40% 里误路由率没有单独测（消融实验只做了整体的），这是可以改进的方向 |

## 第 2 条：三层混合记忆

### 可能问法
- "三层记忆分别解决什么问题？"
- "27.4% 怎么算的？74% 怎么算的？"
- "摘要什么时候触发？多长对话开始压缩？"

### 精简回答（30秒）
三层对应不同粒度：第一层 Redis 存最近 12 轮对话原文，保证即时上下文；第二层超过 12 轮后用 LLM 自动把历史压缩成摘要，让 prompt 不膨胀；第三层 pgvector 存用户级语义记忆，跨会话自动注入。30 轮长对话 token 降 27.4%，跨会话事实召回率 74%。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "27.4% 怎么算的？" | 构造 4 个典型 30 轮对话样本，对比"完整历史塞 prompt" vs "摘要+最近 N 轮"的 token 数，公式 `(full-compressed)/full`，取平均 27.4% |
| "74% 召回率怎么测的？" | 跨会话事实召回。会话 A 注入事实（如青霉素过敏），新会话 B 用自然 query 检索，看 top-5 是否包含目标事实 |
| "Redis 挂了怎么办？" | 降级为纯当前轮对话，不影响核心流程。历史丢失但不会报错 |
| "三层过重了吗？" | 各有分工不冗余。Redis 管即时性、摘要管控 token、pgvector 管跨会话，缺一层都有损失 |

### 指标细节
- **Redis 滑动窗口**：最多 12×2=24 条消息，TTL 24h
- **摘要触发阈值**：`SUMMARY_REFRESH_THRESHOLD=4`，对话超过 12 轮时触发
- **pgvector 检索**：三因子排序 `recency(0.3) × importance(0.4) × relevance(0.3)`，取 top-5

### 为什么有 26% 没召回？（五层漏斗）
```
提取阶段 → LLM 没提取出有用信息
  ↓
保存阶段 → importance < 4 阈值过滤
  ↓
去重阶段 → cosine > 0.9 相似合并（信息变形）
  ↓
排序阶段 → 三因子排序中 importance 权重最高(0.4)，可能把相关但低重要性的挤下去
  ↓
截断阶段 → 只返回 top-5，排名第 6 的就丢了
```

### 瓶颈
- top-5 硬截断，没有自适应 top-k
- 三因子权重是经验值，没有网格搜索优化
- importance 评分依赖 LLM，模型更换可能影响排序
- 没有记忆过期/遗忘机制，永久保留

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"记忆无限增长怎么办？数据库撑爆了？"** | 目前没有过期/遗忘机制，这是已知局限。优化方向：加 `expires_at` 字段，不同类型不同 TTL（medical 类记忆有效期长，decision 类短）；或者用 LLM 定期扫描标记废弃记忆 |
| **"用户同时开两个设备聊天，记忆提取冲突了？"** | 提取是后台异步线程，确实可能两边的 LLM 同时写入。目前靠 `_check_dedup`（cosine > 0.9 合并）缓解，但没有分布式锁。如果真正并发高，需要加 Redis 锁或乐观锁 |
| **"用户想删除某条记忆怎么办？"** | 目前没有暴露删除 API。记忆是系统自动提取的，用户不可见。如果要加，方案是给前端一个"我的记忆"面板，调用 `delete_memory(id)`，同时设 `active=false` 软删除 |
| **"如果用户换了手机号（user_id 变了），记忆能迁移吗？"** | 目前 user_id 是硬关联，换了就丢了。迁移方案：后台批量改 `user_memories.user_id`，或者引入 `external_user_id`（如微信 openid）做稳定标识 |
| **"摘要压缩什么时候触发？如果用户聊了 30 轮但每轮都很短呢？"** | 目前按轮数触发（> 12 轮），不看 token 量。如果每轮很短其实不需要压缩。更好的策略：按实际 token 数触发，超 `BASE_TOKEN_THRESHOLD(4000)` 时压缩 |
| **"Redis 清空了，会话还在 pg 里，能自动恢复吗？"** | 不能。Redis 是即时上下文的热数据，丢了就丢了——降级为"无历史上下文"，但不影响核心流程。pg 里的 user_memories 和 summaries 不受影响 |

## 第 3 条：医疗知识库检索链路

### 可能问法
- "父子块分层索引怎么设计的？"
- "Query Rewrite、RRF、rerank 各解决什么问题？"
- "Precision@5 和 Top-1 命中率怎么测的？"
- "为什么不用 ES/Milvus 而是 pgvector？"

### 精简回答（30秒）
父子块分层：父块 2000-4000 字符做检索粒度，子块 500 字符做展示粒度，检索命中子块回推到父块去重展示。Query Rewrite 把口语化 query 改写为检索友好问题。混合检索（向量 + 全文）用 RRF 融合多路结果，最后 BGE-reranker-v2-m3 重排序。在 NHC/WHO 指南文档集上 Precision@5 从 0.68 → 0.83。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "为什么不用 Milvus？" | pgvector 足够用，而且和用户记忆、病历表在同一个 PG 实例里，省运维。数据量在百万级内 pgvector 性能没瓶颈 |
| "RRF 的 K 值为什么是 60？" | 60 是业界经验值，范围在 0-100 之间不敏感。我们没做网格搜索调优 |
| "rerank 慢不慢？" | BGE-reranker 处理一对 query-doc 约几十 ms。我们粗排取 top-12 再 rerank top-5，总延迟可控 |
| "父子块为什么不用更大的 overlap？" | overlap 太大会增加存储和检索噪音。当前 100 字符够覆盖大多数跨块语义断点 |

### 指标细节
| 指标 | Baseline | 优化后 | 提升 |
|------|----------|--------|------|
| Precision@5 | 0.68 | 0.83 | +22% |
| Top-1 命中率 | 0.61 | 0.79 | +30% |

**测试集**：NHC（国家卫健委）+ WHO 指南文档，构造标准 QA 对，manual 标注 ground truth。

### 设计思路
```
用户 query → Query Rewrite（1-3 个检索友好 query）
  → 多路检索（向量 + 全文）
  → RRF 融合（K=60）
  → 子块回推父块去重
  → Rerank（BGE-reranker-v2-m3）
  → Top-5 返回
```

### 瓶颈
- reranker 是耗时的（比向量检索慢 10-100 倍），查询量上来是瓶颈
- 子块→父块回推可能引入冗余信息
- 没有动态调整检索策略（简单问和复杂问用同一条链路）
- 文档更新依赖定时同步，实时性不够

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"NHC/WHO 文档更新了怎么办？增量还是全量重建？"** | 用 content_hash 做增量。每次同步时算文档 hash，和数据库对比：hash 一致 → 跳过；不一致 → 删旧 chunks 重新索引。避免了全量重建对未变文档的重复 embedding。对比全量同步，时间从 O(全部文档) 降到 O(变更文档) |
| **"用户问的知识库里完全没有怎么办？"** | 这是检索链路上游就能探测到的。`_confidence_bucket()` 返回 `no_evidence` 时，答案生成走降级路径："当前知识库未收录此内容，以下是通用回答..."。不会硬编答案 |
| **"检索返回了结果但全都不相关怎么办？"** | 这是向量检索的"语义漂移"问题——query 和 docs 向量接近但内容不相关。reranker 是主要防线，BGE-reranker 做交叉编码能过滤掉大部分不相关的。如果 reranker 也误判，证据校验节点会拦截 |
| **"父子块切分：如果一条关键信息刚好被切到两个子块中间怎么办？"** | overlay=100 字符就是为这个设计的。关键语句大概率落在 overlap 区域，被两个子块各带一份。即使一个子块没被召回，另一个也有机会命中 |
| **"pgvector 百万级文档后查询变慢怎么办？"** | 当前用了 IVFFlat 索引（`idx_child_chunks_embedding_cosine`），百万级以内够用。再往上可以考虑：① 换成 HNSW 索引（写入慢但查询更快）；② 分区表按 source 分；③ 迁移到专用向量数据库（Milvus/Qdrant） |
| **"如果用户问的是一个非常冷门的罕见病，但知识库有、只是 embedding 匹配不上？"** | Query Rewrite 节点可以缓解——把"戈谢病"改写为"戈谢病 glucocerebrosidase 罕见病 溶酶体贮积症"，增加多个检索关键词。但如果 embedding 模型本身对医学专业术语不敏感，根本解决是换医学微调过的 embedding 模型 |

## 第 4 条：MCP 接入 + LLM 分级 + 熔断

### 可能问法
- "MCP 怎么接多医院的？"
- "120s 熔断是什么机制？"
- "LLM 分级路由怎么分级的？"
- "如果医院从 2 家扩展到 120 家怎么办？"

### 精简回答（30秒）
每家医院独立 MCP 服务进程，工具通过命名空间注入（如 `xiehe__book`），LLM 调用时按前缀区分。每用户凭证 Fernet 加密存数据库。熔断器三态：closed → open → half_open，连续失败 3 次熔断，120s 冷却后探测恢复，每医院独立，互不影响。扩展 120 家时的优化方向是连接池共享 + 熔断器下沉 + 按需连接。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "120s 怎么不做成可配置？" | 可以的，现在硬编码是 mock 环境偷懒。生产环境通过 `MCP_RECOVERY_TIMEOUT` 环境变量暴露 |
| "三态熔断器的 half_open 是怎么探测的？" | 120s 到期后，下一个请求自动放行作为探测请求。成功 → closed，失败 → 回到 open 重新计时。惰性检测，没有定时器线程，在 `allow_request()` 时判断 `time.monotonic()` |
| "如果 MCP 返回了错误但不抛异常怎么办？" | 目前依赖异常机制。生产上需要加返回码校验——非 2xx 也算失败 |
| "用户 token 过期了怎么处理？" | 返回 401 → 熔断器记失败 → 通过 credential_store 的 health 字段标记 |
| "120 家医院扩展怎么设计？" | 拆两个维度：全平台 120 家注册医院（正常）+ 单用户绑定 ≤10 家。优化三点：连接池共享（同一家医院跨用户复用）、熔断器下沉到全局、按需连接（不查不连） |

### 指标细节
- **熔断参数**：`failure_threshold=3`，MCP 层 `recovery_timeout=120s`，LLM 层 `recovery_timeout=60s`
- **LLM 分级**：light 模型（意图路由、简单改写），strong 模型（RAG 答案生成、证据校验）
- **加密**：Fernet（AES-128-CBC + HMAC-SHA256），支持 MultiFernet 密钥轮转

### 设计思路
```
Light LLM ──→ 意图路由、Query Rewrite
                ↓ 如果熔断
              fallback to Strong LLM
                
Strong LLM ──→ RAG 答案生成、证据校验、摘要压缩

MCP 医院：
  协和 MCP ──→ 独立 CircuitBreaker ──→ 挂了不影响仁济
  仁济 MCP ──→ 独立 CircuitBreaker ──→ 挂了不影响协和
```

### 瓶颈
- per-user per-hospital 连接管理，用户量上来后连接数爆炸
- 熔断器也是 per-user per-hospital，无全局感知
- MCP 工具一次性全加载，不区分"常用"和"冷门"
- 没有连接预热机制，首次请求用户等得久

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"120 家医院怎么扩展？"** | 拆两个维度回答。全平台 120 家是注册医院数，单用户绑定 ≤10 家。技术优化三点：① 连接池共享（同医院跨用户复用 Semaphore）；② 熔断器下沉到全局（per-hospital 而非 per-user per-hospital）；③ 按需连接（用户没提的医院不建连）。详见 `mcp-pool-refactor-plan.md` |
| **"所有 MCP 医院全挂了怎么办？"** | 回答分两层。LLM 层面：医院全挂不影响知识问答和科室推荐（这些不依赖 MCP），用户至少还能获得医学建议。产品层面：熔断器全部 open 时，`get_tools_for_user` 返回空列表，底层 RAG 链路不注入任何 MCP 工具，提示词里也不会出现挂号相关选项 |
| **"用户删除了某家医院的凭证，但还有进行中的预约怎么办？"** | 这是数据一致性问题。目前 `invalidate()` 只清理连接池，不管已生成的预约。处理方式：credential_store 标记 `active=false` 软删除，已预约的记录保留到过期自然结束，但新挂号时不会出现该医院选项 |
| **"MCP 握手慢（比如内网延迟大），用户首次对话等太久？"** | 目前 _build_pool 同步阻塞加载所有医院工具。优化：① 预热机制（用户登录后后台异步加载）；② 先加载常用医院，冷门按需加载；③ 单独设 MCP 超时，不阻塞对话主流程 |
| **"工具注入到 prompt 后 token 超限了怎么办？"** | 每个医院注入到 prompt 的工具描述有名字 + 描述 + 参数 schema。120 家医院全注入会导致 prompt 爆炸。两重保护：① 只注入用户绑定的医院（≤10 家）的工具；② 工具描述截断到最长 200 字符；③ 极端情况下按 hospital priority 只取 top-N |

## 第 5 条：预约/取消半受控模式

### 可能问法
- "半受控模式怎么实现的？"
- "为什么不直接用 LLM 判断确认？"
- "30 角色压测 0 误操作是怎么保证的？"
- "checkpoint 中断-续接是什么？"

### 精简回答（30秒）
预约流程三段式：用户说预约 → LLM 提取信息返回预览让用户确认 → 用户确认后代码执行。关键操作不在 LLM 手里在代码手里。LLM 只负责"理解用户说什么"，代码负责"判断能不能执行"。LangGraph 的 `interrupt_before` 在信息不足时自动中断等用户补充，续接时从断点继续。30 个角色的压测覆盖了正常预约、信息不全追问、中途改口、取消等分支，0 误操作。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "0 误操作怎么测的？" | 30 个 Persona（不同年龄、症状、医院偏好），每个跑 10 轮对话，覆盖预约/取消/修改/中断回复，检查最终执行的 API 调用是否和用户真实意图一致 |
| "checkpoint 中断后怎么续接？" | `interrupt_before=["request_clarification"]`，图暂停后等用户下一轮消息。`update_state()` 注入用户消息，图从 `request_clarification` 继续，`route_after_clarification` 根据 `clarification_target` 跳到目标节点 |
| "用户中途改口怎么办？" | `_looks_like_clarification_response()` 检测用户是否真的在回答问题。如果检测到是另起话题（问其他症状、改医院、取消），回到 intent_router 重新路由 |
| "为什么 LLM 不能直接操作预约？" | LLM 生成不稳定的。比如用户说"帮我取消"，但不能 100% 确定取消哪个——可能有多个预约。代码门控保证操作前一定经过"预览→确认" |
| "用户说'是'但没有指定确认什么怎么处理？" | 状态维护了 `pending_action_type` 和 `pending_candidates`，知道上一次问的是什么，所以"是"可以直接关联 |

### 指标细节
- 30 个 Persona，每个 10 轮对话 = 300 个多轮对话样本
- 0 误操作：所有最终执行的预约/取消操作都通过了用户确认检查
- `_looks_like_clarification_response()` 检测了 7 种"不是澄清回复"的情况

### 设计思路
```
预约流程：
  用户说"帮我挂协和的专家号"
    → LLM 解析意图 + 提取医院/科室信息
    → 返回预览：协和医院-心内科-张三医生-6月10日 9:00
    → 用户说"确认"
    
⚠️ 关键点：
  - "确认"之前：LLM 可以参与信息提取
  - "确认"之后：纯代码逻辑，调用 MCP 接口
  - 代码门控：如果没经过确认流程，API 调用被拦截
```

### 瓶颈
- 三段式限制了用户体验（每次都要确认，感觉繁琐）
- 不支持多步骤组合操作（"取消协和的，改挂仁济的"）
- `_looks_like_clarification_response()` 依赖启发式规则，可能有漏判

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"用户确认预约后 MCP 执行失败了怎么办？"** | 这是最关键的边界——确认和执行的原子性。目前 30 角色压测覆盖了这个 case。失败时：① 返回明确错误提示"预约失败：XX 医院当前无法连接"；② 熔断器记录失败，该医院进入冷却期；③ `pending_state` 保留，用户可以重新选择其他医院或稍后重试。预约记录不会被错误创建|
| **"用户在'预览'阶段突然改口说另一个医院/时间怎么办？"** | `_looks_like_clarification_response()` 检测到不是确认回复 → 回到 intent_router 重新路由 → 当成新的预约请求处理。`pending_state` 被覆盖，旧预览失效。不会出现"确认了旧选项"的情况 |
| **"用户说'确认'但系统误解为确认了错误的预约怎么办？"** | `pending_candidates` 里保存了当前待确认的预约列表，确认时校验候选 ID 的一致性。而且确认后的执行走代码门控，不是 LLM 自由文本生成，所以不会出现"LLM 自己编了一个预约号"的情况 |
| **"30 角色压测覆盖了故意捣乱的人吗？"** | Persona 里包含"犹豫型"（反复改口）、"模糊描述型"（信息严重不全）、"极端型"（输入乱码/纯 emoji/bug 报告式的超长文本）。这些 case 的预期行为是触发澄清追问或降级回答，不会误执行预约操作 |
| **"用户说'帮我挂最快的号'——系统怎么处理这种模糊指令？"** | 信息不足 → 触发 `pending_clarification` → 反问"您需要哪个科室？有没有偏好的医院？" → 中断等用户回答。不会自己随机选一个号源 |

## 第 6 条：医疗场景安全兜底

### 可能问法
- "高风险症状怎么判定？"
- "证据校验节点怎么拦截幻觉？"
- "列级加密有什么实际价值？"
- "怎么保证高风险提醒一定会出现？"

### 精简回答（30秒）
安全兜底四条线并行。一是 14 个高风险关键词匹配，命中强制追加线下就医提醒。二是检索证据不足或为空时，回答降级为专业医学信息带免责声明。三是 RAG 输出后过 `answer_grounding_check` 校验节点，LLM 结构化判断答案是否超出证据范围，超出的改写。四是数据库敏感列 Fernet 加密存储，支持密钥轮转。高风险提醒是规则匹配，覆盖率 100%。

### 追问预判
| 追问 | 回答要点 |
|------|---------|
| "14 个关键词够吗？漏判怎么办？" | 14 个覆盖了最常见的急症信号。确实可能漏判（如"突然看不见了"不在列表里），方向是引入 LLM 辅助判断作为规则层的补充 |
| "证据校验节点不会被 LLM 自己绕过吗？" | 校验的是"答案"和"检索文档"的一致性，LLM 不需要"知道真相"只需要比对文本。而且用的 strong LLM + 结构化输出，结果可审查 |
| "Fernet 加密的性能开销多大？" | AES-128 加解密是微秒级，对接口延迟几乎没影响 |
| "如果关键词不在列表但也有风险怎么兜底？" | 这是已知盲区。降级方案：证据校验节点的 prompt 里包含了风险评估提示，LLM 可能提醒。但不是 100% 可靠 |

### 指标细节
- 高风险关键词：14 个（胸痛、胸闷、呼吸困难、呼吸急促、意识模糊、抽搐、大出血、持续高热、晕厥...）
- 证据校验：高置信度时跳过（fast path），低置信度才调 LLM
- 降级分档：`no_evidence` / `low` / `medium` / `high` 四档
- 加密：Fernet（AES-128-CBC + HMAC-SHA256），`enc:v1:` 前缀标识密文

### 设计思路
```
四道防线，层层递进：

  用户 query
    ↓
  【防线 1】_infer_risk_level() → 高风险关键词匹配
    ↓
  RAG 检索 + 答案生成
    ↓
  【防线 2】ground_answer() → 证据不足降级 + 免责声明
    ↓
  【防线 3】answer_grounding_check() → 拦截幻觉答案
    ↓
  输出时：【防线 1】追加高风险提醒（如果有）
    ↓
  存储时：【防线 4】_encrypt_content() → 列级加密
```

### 瓶颈
- 关键词列表是静态的，无法应对新型急症表述
- 证据校验依赖 LLM 评估，LLM 自己也可能误判
- 降级策略是"一刀切"，没有按风险等级细化
- 没有做真实用户场景的覆盖率测试（只在测试集测了）

### 极端情况追问
| 追问 | 回答要点 |
|------|---------|
| **"14 个关键词漏判了怎么办？（比如用户说'突然看不见了'）"** | 这是已知盲区——规则层的短板就是覆盖率 vs 误报率的权衡。两条补充防线：① 证据校验节点检测到检索文档里提到紧急情况时也会追加提醒；② 方向是引入轻量 LLM 做风险辅助判断，但需要平衡延迟和准确率。漏判一个比误判一百个后果严重，所以规则白名单策略偏向保守 |
| **"关键词误判了怎么办？（用户说'我朋友胸痛'被当成高风险）"** | 误判的后果是追一条"请线下就医"提醒，属于信息冗余，没有实际危害。在高风险场景下，宁可误提醒不能漏提醒。而且用户看到提醒可以自行判断——系统只是建议，不做决策 |
| **"证据校验节点自己也幻觉了怎么办？"** | 校验 LLM 的幻觉会导致两个方向的问题——把合理的答案误判为幻觉（过于保守），或把幻觉误判为合理（漏网）。目前用 strong LLM + 结构化输出（Pydantic）来控制输出质量，但没有 100% 可靠。终极方案是人工抽检校验节点的判断结果，发现偏差后调 prompt |
| **"用户问自残/自杀相关的问题怎么办？"** | 目前不在 14 个关键词列表里。需要加"自杀""自残""不想活了""想死"等关键词。回答策略不同——不是"请就医"，而是"请立即拨打心理援助热线 12320/010-82951332"，并提供当地危机干预资源。这个 case 比胸痛更特殊，单纯让就医可能不够 |
| **"用户问的问题涉及法律禁区怎么办？（比如问毒品/安乐死操作方法）"** | 不属于医疗场景兜底，但逻辑类似——加黑名单关键词，命中直接返回固定文案"作为医疗助手，我无法提供此类信息"，不通过 LLM 生成。避免 LLM 在任何约束下被"越狱"输出违规内容 |
| **"降级回答时，用户看不到任何有用信息，体验很差怎么办？"** | 降级不是"什么都不回答"，而是"承认局限性后仍然尽量给一些参考信息"。比如："知识库未收录该罕见病，以下是通用建议：建议您咨询三甲医院专科医生，就诊时带上既往病历..."——给不了专业知识就给出操作指导，保证回答仍然有价值 |

## 全局总结：面试回答心法

### 回答三要素
1. **先讲为什么**（动机）→ 让面试官知道你不是为了用技术而用技术
2. **再讲怎么做**（方案）→ 用最简洁的话说清楚
3. **最后讲验证**（数据）→ 数字比什么都管用
