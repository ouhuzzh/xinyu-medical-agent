## 一、面试本质与核心策略

### 1.1 面试考的不是"你做了什么"，是"你理解了什么"

| 层级 | 面试官在听什么 | 常见误区 |
|------|--------------|---------|
| **做了什么** | 我用了 X 技术 | 罗列技术栈，像读简历 |
| **为什么这么做** | 为什么选 X 而不是 Y | 只说"X 更好"不说好在哪、适合在哪 |
| **还能怎么做** | 有没有想过替代方案 | 没想过，或者只会一种方案 |
| **出了问题怎么办** | 兜底和降级思维 | 只管正常路径，不管异常路径 |

**核心原则**：每个技术点都要能答出三层——**What（做了什么）→ Why（为什么这么选）→ What if（出了问题怎么办 / 替代方案是什么）**。

### 1.2 通用回答框架

```
结论先行 → 实现细节 → 为什么这么选 → 如果不这样会怎样 → 还能怎么优化
```

这个框架适用于所有技术问题。关键是每一步都要有**具体证据**，不是泛泛而谈：
- "延迟降低了" → 不如 "意图分类从 LLM 的 ~200ms 降到规则的 ~5μs"
- "检索更准了" → 不如 "Precision@5 从 0.68 提升到 0.83"
- "更安全了" → 不如 "关键操作走代码校验不经过 LLM，prompt injection 无法绕过"

### 1.3 数字是你的弹药

面试中能脱口而出的数字会大幅提升可信度。提前准备好以下维度的数字：

| 维度 | 示例指标 |
|------|---------|
| 性能 | 延迟、QPS、响应时间 |
| 准确性 | Precision@K、准确率、召回率 |
| 规模 | 数据量、用户量、文档数 |
| 收益 | Token 降低率、成本节约 |
| 设计参数 | 窗口大小、阈值、超时时间 |

**注意**：数字必须诚实。小样本就承认小样本，离线测试就说是离线测试。"10 个样本上的 Precision@5=0.83" 比 "准确率很高" 更专业，也比虚报大样本数据更安全。

## 二、面试风格识别与应对

### 2.1 两种核心面试风格

不同轮次/不同面试官风格差异巨大。识别风格后调整策略，比用同一套打法更有效。

| 维度 | 深挖型（常见二面/技术面） | 宏观型（常见三面/总监面） |
|------|------------------------|------------------------|
| **关注点** | 怎么实现的 | 为什么这么做、还能怎么做 |
| **提问风格** | 连环追问挖细节 | 宏观问题看视野 |
| **深度方向** | 到函数级、代码级 | 到架构级、决策级 |
| **新问题** | 少，围绕项目深挖 | 多，开放性场景题 |
| **软素质** | 不怎么考 | 重点考察 |
| **典型问题** | "你这个规则优先级怎么定的？" | "如果让你从零设计，你会怎么做？" |

### 2.2 应对深挖型面试

**核心策略**：不怕追问，追到底就答到底。答不上来说"我当前方案没覆盖这个场景，如果要做我会考虑 XXX"——这比硬编一个答案好。

**追问模式**：What → Why → What if → Trade-off

| 层级 | 面试官在想 | 你的回答策略 |
|------|-----------|-------------|
| What | 你真的做了吗？ | 具体到代码、文件、函数名 |
| Why | 你理解为什么这么做吗？ | 对比替代方案，说清取舍 |
| What if | 你考虑过边界吗？ | 给出兜底方案和降级路径 |
| Trade-off | 你有判断力吗？ | 承认不足，给出优化方向 |

### 2.3 应对宏观型面试

**核心策略**：从具体项目跳到一般性思考，再回到项目验证。回答结构：**观点 → 项目中的实例 → 一般性规律/方法论**。

| 层级 | 示例 |
|------|------|
| 观点 | "RAG 的核心挑战是检索和生成割裂" |
| 项目实例 | "我的项目中检索错了生成一定错，所以加了充足性检查自动重试" |
| 一般性规律 | "未来方向是检索和生成的深度耦合，比如 FLARE、Self-RAG" |

## 三、项目介绍：3 分钟讲清你的价值

### 3.1 结构模板

```
1. 一句话说清项目解决了什么问题（用户视角，不是技术视角）
2. 整体架构一句话（前端X + 后端Y + 核心Z）
3. 最核心的设计决策 + 为什么（1-2个，展开讲）
4. 量化的成果（2-3个数字）
5. 最近的工程化加固（展示你在往生产方向走）
```

### 3.2 常见错误

| 错误 | 问题 | 修正 |
|------|------|------|
| 从技术栈开始 | 面试官不关心你用了什么，关心你解决了什么 | 从用户痛点开始 |
| 罗列所有功能 | 没有重点，3分钟讲不完 | 只讲 1-2 个最核心的设计决策 |
| 只讲做了什么 | 看不出你的思考深度 | 每个点带一句"为什么这么做" |
| 用技术黑话 | "我用了 LangGraph 做状态机" → 面试官可能不懂 | "系统用状态机管理多意图流转，保证用户中途切换不丢状态" |
| 没有数字 | "效果不错" → 没有说服力 | "意图分类准确率约 95%，规则层延迟微秒级" |

### 3.3 一句话项目描述的公式

**[用户痛点] + [核心方案] + [关键结果]**

- ❌ "我做了一个医疗 RAG 系统"
- ✅ "解决了医疗场景下问诊、分诊、挂号三个需求在一轮对话中无缝切换的问题，用规则+LLM 两阶段路由实现 95% 意图分类准确率"

### 3.4 项目自评：展示自省能力

面试官可能问"给你的项目打几分"。回答公式：

```
打 X 分。扣掉的分数在三个地方：
1. [评估体系] —— 具体缺什么
2. [生产可靠性] —— 具体缺什么
3. [闭环机制] —— 具体缺什么
```

**关键**：不是谦虚，是展示你知道"做完"和"做好"的差距。每个扣分点都要能说出具体的改进方案。

## 四、技术深度：三层追问法

### 4.1 架构与核心链路

**必考题**：画出系统架构图，讲清一个完整请求链路。

**回答策略**：

1. **先画大框架**——不要上来就画细节，先画请求从用户到响应的主链路
2. **标注关键决策**——每个模块旁边标注"为什么用这个方案"
3. **讲清一个完整链路**——从用户发消息到收到回复，每一步走到哪个模块
4. **准备追问**——"如果中途用户岔开话题呢？""如果某个模块挂了呢？"

**通用架构描述模板**：

```
[用户端] → [API 层：鉴权+限流] → [核心引擎：状态机/路由] → [三条处理链路]
                                                              ↓
                                              ┌───────────────┼───────────────┐
                                              ▼               ▼               ▼
                                          [链路A]          [链路B]          [链路C]
                                              ↓               ↓               ↓
                                          [存储A]          [存储B]          [存储C]
```

### 4.2 核心模块深挖

每个核心模块都要准备以下问题的答案：

| 问题 | 回答要点 |
|------|---------|
| 为什么用这个方案？ | 对比 2-3 个替代方案，说清取舍 |
| 为什么不全用 X？ | 给出 X 的三个具体缺点 |
| 具体怎么实现的？ | 到函数级、代码级 |
| 边界 case 怎么处理？ | 举 1-2 个真实边界 case |
| 如果 X 挂了怎么办？ | 降级/兜底方案 |
| 如果要扩展怎么做？ | 当前架构的可扩展性评估 |

### 4.3 数据存储：为什么选这个？

这是高频追问方向。每个存储选型都要能说清：

| 维度 | 要回答的问题 |
|------|-------------|
| 数据量 | 当前多少？预期增长？到什么量级要换？ |
| 性能 | 读写延迟？QPS 上限？ |
| 运维 | 部署复杂度？监控？备份？ |
| 一致性 | 事务需求？跨系统同步？ |
| 替代方案 | 为什么不用 XXX？什么场景下该换？ |

**通用原则**：不要说"XXX 更好"，要说"在当前约束下 XXX 更合适"。好的技术选型不是选最强的，是选约束条件下最合适的。

### 4.4 通信方式选择

| 方案 | 适用场景 | 不适用场景 |
|------|---------|-----------|
| SSE | 服务端单向回流 token/状态 | 高频双向通信 |
| WebSocket | 高频双向实时通信 | 简单请求-响应、代理层不友好 |
| gRPC | 微服务内部同步调用 | 浏览器直连 |
| MQ | 异步解耦、削峰 | 低延迟同步场景 |

**回答格式**："我选 SSE 因为场景本质是客户端发一次请求、服务端持续单向回流，SSE 语义更贴合。WebSocket 虽然更强大但我的场景用不到双向通信，反而增加了复杂度。"

## 五、系统设计：从项目级到产品级

### 5.1 规模化升级

面试官可能问"如果用户量涨 10 倍 / 日活 10 万，你怎么改？"

**通用回答框架——三个核心差异**：

| 维度 | 项目级 | 产品级 |
|------|--------|--------|
| **架构** | 单体应用 | 微服务拆分（API 网关 + 核心服务 + 独立存储服务） |
| **存储** | 单点/同库 | 分层（Redis 集群 + DB 读写分离 + 专业向量库 + CDN 缓存） |
| **可观测性** | 日志 | 完整（链路追踪 + 指标监控 + 告警 + 质量监控） |

**关键**：说完差异后，要说清**什么是不变的**——核心架构设计通常是可扩展的，不需要推倒重来。这展示你的架构前瞻性。

### 5.2 多租户设计

如果面试官追问多租户，三个核心改动：

1. **数据隔离**——按 `tenant_id` 过滤（逻辑隔离）或独立 schema/实例（物理隔离）
2. **配置可定制**——每个租户的规则、流程、UI 可配置
3. **权限模型升级**——user / tenant_admin / platform_admin 三级

### 5.3 加新功能的扩展性评估

回答框架：

1. **当前加新功能要改哪些文件**——具体列出来
2. **问题在哪**——改动散落在多处、没有编译期保护、加新容易改坏旧
3. **优化方案**——注册机制/插件化/配置化
4. **为什么现在没做**——当前规模下维护成本可接受，但边际成本在上升
5. **什么条件下必须做**——给出具体阈值（比如"意图超过 7-8 个"）

## 六、生产思维：故障与降级

### 6.1 故障排查通用框架

面试官问"线上出了 XX 问题怎么排查"，按以下框架回答：

```
1. 定位瓶颈层 —— 是网络/API/计算/存储/外部依赖？
2. 看监控指标 —— 每层的延迟、错误率、资源利用率
3. 链路追踪 —— 请求经过哪些节点，哪个节点慢
4. 日志分析 —— 异常日志、慢查询日志
5. 复现验证 —— 确认根因，不是猜测
```

### 6.2 关键依赖故障应对

对每个外部依赖（LLM、Redis、数据库、向量库），都要准备好：

| 问题 | 回答要点 |
|------|---------|
| 挂了系统还能工作吗？ | 哪些功能可用、哪些不可用 |
| 有降级方案吗？ | 具体 fallback 策略 |
| 降级后数据一致性怎么保证？ | 恢复后的状态重建 |
| 有监控和告警吗？ | 健康探针、熔断阈值 |

**LLM 降级链**（通用模板）：

```
主 provider → 备用 provider → 规则兜底 → 安全提示
```

**Redis 故障策略**：

| 环境 | 策略 | 原因 |
|------|------|------|
| 开发 | 降级到进程内存 | 优先可用性，方便调试 |
| 生产 | fail fast，启动失败 | 多实例下降级会导致会话不共享、状态错乱 |

### 6.3 接口变慢排查

通用排查路径：

1. **链路追踪**——哪个环节慢（API 路由 / 业务逻辑 / 外部调用 / 缓存 / 数据库）
2. **外部依赖**——LLM provider 延迟是否突增
3. **缓存**——Redis 慢查询、缓存命中率
4. **数据库**——慢查询、执行计划、索引是否生效
5. **向量检索**——是否走了索引、数据量增长后是否需要调参

## 七、设计权衡：工程判断力

### 7.1 最满意的技术决策

**回答公式**：

```
决策名称 + 为什么不是中庸而是最优 + 边界定义的难点 + 你怎么定边界
```

不要说"因为它最复杂"或"因为它最新"，要说"因为它最好地匹配了当前约束"。

**示例结构**：
- "规则+LLM 两阶段路由"
- 不是中庸，是把两种方案的优势对准了不同场景
- 难点在边界定义——规则太宽误路由，太窄 LLM 压力大
- 策略：规则宁可窄（高精确率），漏的交给 LLM（高召回率），因为误路由代价远大于多处理一个 query

### 7.2 最后悔的技术决策

**回答公式**：

```
缺了什么 + 如果重来怎么做 + 你从中学到什么
```

**注意**：不要说"没什么后悔的"——这要么说明你没反思，要么说明项目太简单。

常见后悔方向：
- 没做消融实验/量化评估
- 规则硬编码没有配置化
- 没做 LLM 熔断和降级
- 没有用户反馈闭环

### 7.3 怎么判断"够用"还是"需要优化"

两个维度：

1. **当前指标是否满足业务要求**
2. **指标恶化速度是否可控**

如果当前够但增长后会恶化 → "够用"只是暂时的，提前规划
如果当前不够但优化成本极高 → 评估"不够"对业务的影响，再定优先级

**核心原则**：**不为优化而优化，为业务目标而优化**。不同场景优先级不同——医疗安全和准确性优先，通用场景可能延迟和成本优先。

### 7.4 常见技术选型的回答模板

每个选型都要能回答：

```
1. 当前方案：是什么
2. 替代方案：有哪些（至少 2 个）
3. 当前方案的优势：在当前约束下为什么更合适
4. 当前方案的局限：什么时候不够用
5. 迁移条件：什么条件下该换方案
```

## 八、技术视野与行业认知

### 8.1 RAG 方向演进（通用知识）

面试官可能问"你觉得 RAG 未来会怎么演进"。

**三个方向**：

1. **检索和生成的深度耦合**——FLARE（生成中动态检索）、Self-RAG（模型自主决定是否检索）
2. **长上下文对 RAG 的影响**——上下文够长是不是可以不要 RAG？大多数场景不行：知识库持续增长、成本太高、注意力衰减
3. **多模态 RAG**——影像、表格、图表的检索和融合

**回答技巧**：讲完行业方向后，一定要回到你的项目——"我当前方案的局限是 XXX，未来如果要演进，方向是 YYY"。

### 8.2 Agent 方向认知

**Agent 的核心三要素**：自主决策 + 工具调用 + 环境交互

**回答思路**：

| 问题 | 回答策略 |
|------|---------|
| 你的项目算不算 Agent？ | 说清是"受控 Agent"——LLM 负责理解和生成，关键路径由代码控制 |
| 为什么不做全自主 Agent？ | 安全场景下风险太高，模型可能走错路径 |
| 未来怎么做？ | 场景更复杂时需要更自主的 Agent，但安全机制必须同步升级 |

**行业认知**：Anthropic 的 Constitutional AI 思路——给 AI 设边界，而不是给完全自由。

### 8.3 LLM Provider 选择

五个维度：

1. **效果**——目标场景下的实际能力，不能只看 benchmark
2. **延迟**——TTFT（Time To First Token），对话场景用户敏感
3. **成本**——分级使用：高频低价值用便宜模型，低频高价值用强模型
4. **稳定性**——SLA、降级方案、多 provider 支持
5. **合规**——数据出境、行业监管

### 8.4 你对技术趋势的看法（通用回答框架）

```
1. 当前主流方案的核心局限是什么
2. 行业在往什么方向演进（2-3 个方向）
3. 每个方向的挑战和机遇
4. 你的项目在什么位置，怎么跟上
```

## 九、软素质与成长潜力

### 9.1 最大的困难

**回答公式**：

```
不是"技术难"，是"选择难"
→ 具体的困境描述
→ 转变思路的关键认知
→ 这个认知如何影响后续决策
```

**示例**：意图冲突的边界定义——不是匹配难，是定义难。转变思路从"规则覆盖一切"到"规则保核心、LLM 兜边界"。关键认知是**规则误路由的代价远大于 LLM 多处理一个 query 的代价**。

### 9.2 团队协作与冲突解决

**回答公式**：

```
冲突是什么 → 为什么产生 → 怎么解决 → 结果如何
```

**关键**：解决冲突的方法要说清——**说清为什么 + 降低对方的改动成本**。不要只说"我说服了他"，要展示你的沟通方法。

### 9.3 核心差异

**不要说**："我更努力""我学习能力强"——太虚了。

**要说**：具体的能力差异，最好有对比：

1. **端到端系统思维**——理解模块间依赖比做好单个模块更重要
2. **技术选型有判断标准**——每个决策都有"为什么这么选"和"不这么选会怎样"

### 9.4 学到了什么（认知层面）

三层转变：

| 层次 | 从 | 到 |
|------|----|----|
| **技术能力** | "会用" | "会选"——从"会调 API"到"知道什么该交给模型什么不该交" |
| **工程思维** | "能不能跑" | "出了问题怎么办"——每个设计决策都问"如果这个环节失败了怎么办" |
| **认知转变** | "追求最优" | "追求合适"——没有最优方案，只有约束条件下最合适的方案 |

**一句话总结**：从"把功能做出来"到"把功能做可靠"。做出来靠技术能力，做可靠靠工程判断。

### 9.5 未来方向

**回答原则**：

1. 和项目经验一致——不要说一个和你项目无关的方向
2. 有具体子方向——不是泛泛的"AI 方向"
3. 说清核心挑战——不是"我想学 XX"，是"我想解决 XX 问题"

**示例**：AI 应用工程方向——把 LLM 的能力转化为可靠的产品。核心挑战不是"模型能不能做到"，是"做到了怎么保证可靠、可控、可观测"。

## 附录 B：高危追问速答模板

> 压力追问下 15-30 秒快速答稳。核心原则：**承认局限 > 硬编答案**。

### B.1 通用高危问题

#### "你这个数字靠得住吗？"

> 说清测法、样本规模、适用边界。主动补一句"这个结果主要用于指导方案选择，如果要做更强结论还需要更大测试集和消融实验"。诚实比虚报更稳。

#### "你这算生产可用吗？"

> 更准确的说法是"可部署原型，已经做了工程化加固，但还不是完整生产系统"。列出已做的和还缺的，比直接说"能上生产"更可信。

#### "为什么不选 XXX（更流行的方案）？"

> 三步回答：①承认 XXX 的优势 ②说清当前约束下为什么不选 ③什么条件下该换。

#### "LLM 判断错了怎么办？"

> 关键操作不能依赖 LLM 判断，走代码级门控。LLM 可以被 prompt 影响，代码不会。

#### "状态不会丢吗？"

> 说清持久化机制和恢复路径。不是"不会丢"三个字就够了，要说出具体机制。

#### "你这不是过度工程了吗？"

> 说清当前规模下每个设计的必要性。如果某个设计在当前规模下确实是过度工程，大方承认"这个确实是提前做了，因为 XXX"。

#### "如果让你重新做，会改什么？"

> 这是加分题不是减分题。能说出具体的改进点 = 你有反思能力。说"没什么要改的"才是真正的减分。

### B.2 项目特定高危问题速答

> 以下以医疗智能助手为例，替换成你自己的项目。

| 问题 | 速答要点 |
|------|---------|
| 为什么不全用 LLM？ | 延迟、成本、确定性三维度 |
| 规则多了维护不了怎么办？ | 规则只覆盖高频确定性场景，模糊的交给 LLM，数量有上限 |
| 怎么防 prompt injection？ | 三层：意图路由硬逻辑 + 确认门控代码校验 + grounding 检查 |
| LLM 幻觉怎么办？ | 两层：检索阶段要求只用检索结果 + 生成后 grounding check |
| 怎么防止误预约？ | 确认词代码硬校验，不经过 LLM |
| 中途岔开话题怎么办？ | pending 状态保留，新话题走正常路由，回来可续接 |
| Redis 挂了怎么办？ | 开发降级，生产 fail fast |
| 向量检索慢怎么办？ | 检查索引、维度、数据量、查询计划 |
| 用户量涨 10 倍呢？ | LLM 推理排队 + 向量库迁移 + Redis 集群 |
