# 医疗智能助手项目面试准备

## 1. 项目一句话介绍

基于 LangGraph 搭建的医疗智能助手，支持医疗知识问答、跨会话记忆、科室推荐、预约挂号和取消预约，并具备多医院 MCP 接入、Skill 插件化扩展、混合检索和关键操作安全控制能力。

适合开场的 30 秒版本：

> 这是一个偏工程化的 Agent 项目，不是单轮 RAG Demo。我主要做了四块：第一是声明式 Skill 插件化路由，解决新增能力时核心流程不断膨胀的问题；第二是 Redis、摘要和 pgvector 组成的三层记忆，解决长对话成本和跨会话记忆问题；第三是父子块、混合检索、Query Rewrite、RRF 和 rerank 组成的医疗 RAG 链路，提升证据命中率；第四是多医院 MCP 接入和预约半受控执行流程，保证真实工具调用的隔离性和安全性。

适合开场的 60 秒版本：

> 这个项目是一个基于 LangGraph 的医疗智能助手，面向真实医疗咨询场景，覆盖知识问答、跨会话记忆、科室推荐、预约挂号和取消预约。我的核心工作有四部分。第一是把意图识别做成声明式 Skill 插件化路由，Skill 通过关键词、语义示例和 LLM 提示接入规则、语义、LLM 三层分类流程，新增同类意图时不需要改核心路由框架。第二是设计三层混合记忆，用 Redis 保存最近几轮对话，用摘要压缩长历史，用 pgvector 保存用户级长期事实，解决长对话 token 成本和跨会话记忆问题。第三是优化医疗知识库检索，使用父子块索引、混合检索、Query Rewrite、RRF 和 rerank，提高指南类问题的命中率。第四是对接多医院 MCP 工具，设计凭证隔离、工具命名空间、熔断器和预约确认机制，避免单医院故障和误调用风险。

3. 有的旧文档把规则覆盖率、延迟收益等数字写得比较具体，但这些数字在当前仓库里不一定都有稳定依据。

5. 旧文档里有些实现描述偏早期版本，比如把某些能力讲成硬编码链路，而现在项目里已经更偏 Skill Registry 和节点化路由。

建议你复习时遵循这个优先级：

1. 当前源码和 README
2. 本文档
3. 旧面试资料里的追问题目

## 3. 面试总体主线

面试这类项目，尽量不要被面试官带成“背技术名词”。主线要始终回到四个工程问题：

1. 能力扩展问题：新增意图和工具时，核心流程会不会越来越乱。
2. 上下文成本问题：长对话会不会越来越贵，跨会话会不会完全失忆。
3. 检索质量问题：医疗问题对证据质量要求高，普通向量检索够不够。
4. 工具安全问题：预约、取消这类真实操作，怎么防止误调用和单点故障。

把每一条项目经历都讲成“问题 -> 方案 -> 效果”，整体会很稳。

## 4. 项目条目逐条拆解

### 4.1 Skill 插件化意图路由

#### 这条解决什么问题

传统路由很容易写成很多 `if-elif`，新增一个意图就要改主流程，后面会越来越难维护。另一个问题是纯规则不够鲁棒，用户换个说法就识别不到；但如果所有请求都走大模型，成本和延迟又会很高。

#### 你项目里是怎么做的

把每个能力抽象成一个 Skill。每个 Skill 声明：

1. `keywords`：给规则层用。
2. `utterances`：给语义匹配层用。
3. `llm_hint`：给 LLM 分类层用。
4. `route target`：这个意图最终要跳到哪个节点。

三层路由逻辑：

1. L1 规则层：关键词高置信命中，最快最便宜。
2. L2 语义层：embedding 相似度匹配，解决换说法问题。
3. L3 LLM 层：前两层不确定时，结合上下文做结构化分类。

另外，LangGraph Checkpoint 会保存状态，这样用户补充信息后，不需要整个流程从头跑。LLM 分级路由则让轻量任务尽量交给轻模型，把重模型留给关键节点。

#### 亮点怎么讲

亮点不只是“做了意图识别”，而是把意图识别做成了可扩展、可插拔、可分层降本的路由系统。

#### 推荐回答

> 我做了一个声明式 Skill 插件化意图路由。核心出发点是普通 Agent 项目里的意图识别很容易写成一堆 if-else，新增能力时要不断改核心流程，而且纯规则不够稳、全走大模型又太贵。所以我把每个 Skill 抽象成配置单元，让它声明关键词、语义样例和 LLM 提示，然后统一接入规则、语义、LLM 三层分类。高确定性的请求优先走规则，模糊表达走语义，相对复杂的情况再交给 LLM。新增同类意图时，主要补 Skill 定义和节点注册，不需要改核心路由框架。另外我结合 LangGraph 的 Checkpoint 和轻重模型分级路由，减少了重复推理和不必要的大模型调用。

#### 面试官可能会问

1. 为什么不直接全用 LLM 分类？
2. 为什么不只用规则？
3. 语义层具体怎么做？
4. 所谓“声明式”到底声明了什么？
5. 你说新增意图不用改核心代码，这个说法会不会太满？
6. Checkpoint 为什么能降低成本？
7. LLM 分级路由是怎么分的？

#### 稳妥回答

1. 不全走 LLM，因为简单请求用规则就够，全部走大模型会增加延迟和成本。
2. 不只用规则，因为用户表达不稳定，纯关键词覆盖不全。
3. 语义层通过 Skill 配置的典型 utterances 做 embedding 匹配，处理“没命中关键词但语义明显属于某意图”的情况。
4. 声明的是识别信息和路由信息，不是把逻辑散落在主流程里。
5. 更准确的说法是：新增同类型意图时，不需要改核心路由框架；如果是全新执行模式，仍可能新增 handler。
6. Checkpoint 保存图状态，补充信息时从中间继续，不需要每轮从头推断。
7. 意图分类、摘要、query rewrite 这类轻任务用轻模型，答案生成和复杂工具理解用强模型。

#### 容易被问倒的点

不要把“零改核心代码”讲得绝对化，建议自己口头上改成“新增同类意图时通常不需要改核心路由框架”。

### 4.2 三层混合记忆

#### 这条解决什么问题

长对话时，历史消息如果全部原样拼进 Prompt，token 成本会越来越高。另一方面，用户的病史、过敏史、偏好如果不能跨会话记住，医疗助手就很难做得像一个持续服务用户的产品。

#### 你项目里是怎么做的

三层记忆分别对应不同时间尺度：

1. Redis 滑动窗口：保存最近几轮原始消息，解决短期上下文连贯问题。
2. LLM 摘要压缩：把更长的会话历史压缩成摘要，降低 Prompt 长度。
3. pgvector 长期记忆：保存用户级稳定事实，比如过敏史、慢性病、偏好，用于跨会话召回。

长期记忆不是所有对话都存，而是只保留稳定、重要、可复用的信息，并按相关性、重要性、时间因素综合召回。

#### 亮点怎么讲

亮点在于把“记忆”从简单拼聊天记录，做成按生命周期分层管理的上下文系统。

#### 推荐回答

> 这个项目里我做了一个三层混合记忆架构。因为医疗助手一方面有长对话场景，如果把所有历史消息都塞进 Prompt，成本会不断升高；另一方面又需要跨会话记住用户的过敏史、病史和偏好。所以我把记忆拆成三层：Redis 保存最近几轮原始消息，保证当前会话连贯；摘要层压缩更长的会话历史，降低 token 消耗；pgvector 保存用户级长期语义记忆，用于跨会话召回稳定事实。这样既控制了上下文长度，又保留了个性化和医疗相关的重要信息。

#### 面试官可能会问

1. 为什么一定要三层，不能只存聊天记录？
2. Redis、摘要、pgvector 分别适合什么场景？
3. 什么信息会写入长期记忆？
4. 怎么避免记忆污染？
5. 为什么长期记忆适合用向量检索？
6. Token 降低 27.4% 是怎么测的？
7. 事实召回率 74% 怎么定义？

#### 稳妥回答

1. 只存聊天记录只能解决短期连贯，解决不了长对话成本和跨会话失忆问题。
2. Redis 处理最近几轮，摘要处理本会话长历史，pgvector 处理跨会话稳定事实。
3. 优先写入过敏史、慢性病、药物相关、偏好、决策型信息。
4. 通过预过滤、重要性阈值、矛盾检测和按类型筛选来控制污染。
5. 因为后续问题常常不是原词复现，而是换一种说法，需要语义召回。
6. 用固定多轮样本，对比原始全历史拼接和三层记忆策略下的 Prompt Token 消耗。
7. 在跨会话测试样本里，统计本应被召回的关键事实中，最终成功被检索并带入上下文的比例。

#### 容易被问倒的点

别说“所有用户信息都会记住”，那会引出隐私和污染问题。要强调“只存稳定、高价值、可复用的信息”。

### 4.3 医疗知识库检索链路优化

#### 这条解决什么问题

医疗问答对证据质量要求很高。只做简单向量检索，常见问题包括：

1. 用户提问很口语，和指南原文表达差异大。
2. 向量检索能召回语义相似，但不一定擅长精确术语。
3. 小块检索命中了，但上下文不完整，生成时容易断章取义。
4. 初召回结果排序不稳定，Top-1 不一定是最适合的证据。

#### 你项目里是怎么做的

完整链路可以拆成五步：

1. Parent-child chunking：小块检索，大块生成。
2. Query Rewrite：把口语问题改成更适合检索的表达。
3. Hybrid retrieval：向量检索 + 关键词检索。
4. RRF：把多路召回结果融合。
5. Rerank：对候选文档做精排。

#### 每个词分别解决什么问题

1. 父子块索引：
   子块粒度小，检索更准；父块上下文更完整，生成更稳。
2. 混合检索：
   向量检索擅长语义，关键词检索擅长精确术语，两者互补。
3. Query Rewrite：
   解决用户表达口语化、模糊化的问题。
4. RRF：
   解决不同检索方式分数尺度不一致的问题，用排名融合更稳。
5. Rerank：
   解决初召回里“相关但排序不够准”的问题，让最相关证据尽量排到前面。

#### 亮点怎么讲

亮点在于不是简单“上了 RAG”，而是围绕检索准确性做了多阶段优化，尤其适合证据敏感的医疗场景。

#### 推荐回答

> 医疗问答比普通 FAQ 更依赖证据质量，所以我没有只做单路向量检索，而是搭了一条更完整的检索链路。首先用父子块索引解决检索精度和上下文完整性的矛盾，小块用于召回，父块用于生成。然后用 Query Rewrite 把用户口语问题改写成更适合检索的表达。召回阶段同时做向量检索和关键词检索，兼顾语义和术语匹配，再用 RRF 融合多路结果，最后用 rerank 做精排，提升 Top-1 和 Precision@5。

#### 面试官可能会问

1. 为什么不只做向量检索？
2. 父子块索引具体怎么切？
3. 为什么 Query Rewrite 有必要？
4. RRF 是什么，为什么不用直接拼分数？
5. Rerank 带来的收益是什么，代价是什么？
6. Precision@5 和 Top-1 是怎么定义的？
7. 为什么你的 README 和简历里 Top-1 有时写 0.79，有时写 0.80？

#### 稳妥回答

1. 向量检索对语义有优势，但医疗术语、药名、指南名这类精确词经常要靠关键词检索兜底。
2. 先按文档结构切较大的父块，再继续切小子块；检索命中子块后按 `parent_id` 回取父块。
3. 因为用户常常问“这个严重吗”“这种情况怎么办”，而知识库通常是更规范的医学表达。
4. RRF 按排名融合，不依赖不同检索器的原始分数可比性，所以更稳定。
5. Rerank 主要提升排序质量，代价是额外延迟，所以通常放在较小候选集上做。
6. Precision@5 看前五条里相关证据比例，Top-1 看第一条是否命中目标证据。
7. 口径上建议你统一成一版再讲，如果按照当前 README，优先说 0.79；如果简历写 0.80，口头上可以说“四舍五入后接近 0.80”。

#### 容易被问倒的点

你仓库里 `README` 和简历草稿的指标有一处不完全一致，面试前最好统一口径。

### 4.4 基于 MCP 的多医院接入

#### 这条解决什么问题

如果不同医院的挂号系统都单独接一套接口，Agent 层会很快被各种医院差异拖垮。另一个问题是，某一家医院挂了，不能把整个系统都拖死。再一个问题是，不同用户、不同医院的凭证必须隔离，不能串。

#### 你项目里是怎么做的

1. 用 MCP 协议把医院工具能力抽象成统一接口。
2. 为每个用户维护自己的医院工具池。
3. 用户凭证加密存储，运行时按用户解密加载。
4. 工具名做命名空间隔离，避免多医院工具重名。
5. 每家医院单独挂一个熔断器，失败达到阈值后短时隔离，120 秒后再尝试恢复。

#### 亮点怎么讲

亮点是把多医院接入问题从“多套接口拼装”抽象成“统一协议 + 用户级隔离 + 医院级故障隔离”。

#### 推荐回答

> 多医院接入的核心难点不只是把工具调通，而是要解决统一抽象、凭证隔离和单点故障隔离。我这边用 MCP 协议把不同医院的挂号能力统一成标准化工具接口，Agent 不需要感知底层医院实现差异。同时每个用户都有自己的医院工具池，凭证加密存储、按用户解密加载，避免串用。工具名也做了命名空间隔离，避免不同医院同名工具冲突。另外每家医院都挂三态熔断器，某一家医院连续失败时会被临时摘除，但其他医院仍然能正常提供服务。

#### 面试官可能会问

1. MCP 的价值是什么？
2. 为什么要做工具命名空间？
3. 凭证隔离是怎么做的？
4. 熔断器三态是什么意思？
5. 为什么恢复时间设成 120 秒？
6. 单医院故障为什么不会影响其他医院？

#### 稳妥回答

1. MCP 让多医院工具可以按统一协议接入，减少 Agent 层对医院差异的耦合。
2. 不同医院可能存在同名工具，不做命名空间会冲突，也不利于定位调用来源。
3. 凭证按 `(user_id, hospital_code)` 维度加密存储，运行时只加载当前用户绑定的医院凭证。
4. 正常、打开、半开，半开阶段用于探测是否恢复。
5. 120 秒是工程上的折中，既避免持续打挂故障服务，也不会让恢复等待过长。
6. 因为每家医院的工具加载、健康状态和熔断都是独立维护的。

#### 容易被问倒的点

不要把“协议统一”讲成“所有医院接口完全一样”。更准确的说法是：对 Agent 暴露的是统一工具协议，底层仍可能有适配差异。

### 4.5 预约/取消半受控执行流程

#### 这条解决什么问题

预约和取消属于有真实后果的操作。不能让 LLM 理解完意图后直接执行，否则很容易误操作。

#### 你项目里是怎么做的

完整流程大致是：

1. 用户提出预约或取消需求。
2. 系统收集参数，比如科室、日期、时段、医生或预约号。
3. 先生成预览，不真正执行。
4. 把待执行动作写入 pending state。
5. 用户明确确认后，代码层才真正调用执行接口。

也就是说，LLM 负责理解和整理，不直接拥有关键操作执行权。

#### 亮点怎么讲

亮点在于把真实工具调用做成了“预览 -> 确认 -> 执行”的半受控状态机，而不是交给模型直接拍板。

#### 推荐回答

> 预约和取消是有真实后果的动作，所以我设计成半受控流程。LLM 只负责理解用户意图和整理参数，系统会先生成预约或取消预览，比如科室、日期、时段、医生、预约号等，并把待执行动作放进 pending 状态。只有用户明确回复“确认预约”或“确认取消”这类确认语义后，代码层才允许真正调用执行接口。这样能有效避免 LLM 误解指令后直接提交。

#### 面试官可能会问

1. 为什么不让模型直接执行？
2. 如果用户中途改时间怎么办？
3. 怎么处理“取消”这个词的歧义？
4. 0 误操作是怎么压测出来的？
5. 如果确认时号源没了怎么办？

#### 稳妥回答

1. 因为真实工具操作必须有确定性的状态控制，不能只依赖模型理解。
2. 在 pending 状态下允许继续修改参数，系统更新 payload 并重新预览。
3. 结合当前上下文判断，是取消本次待确认动作，还是取消已有预约。
4. 用多角色脚本或 persona 压测覆盖预约、取消、打断、澄清等组合场景，看是否出现未经确认直接执行。
5. 确认时会再次校验可用性，如果资源失效就提示重新选择，而不是静默失败。

#### 容易被问倒的点

不要说“完全不会误操作”，更稳的表达是“在当前 30 角色压测样本下未观察到误操作”。

### 4.6 安全兜底机制

#### 这条解决什么问题

医疗场景不能只追求回答像不像，还要考虑风险控制。典型风险包括：

1. 高风险症状被轻描淡写地回答。
2. 检索证据不足时模型胡乱发挥。
3. 用户隐私和敏感医疗信息泄露。

#### 你项目里是怎么做的

1. 高风险症状识别后触发就医提醒。
2. 证据不足时降级为一般医学信息回答，并附带免责提示。
3. 回答前做 grounding / evidence check，尽量抑制幻觉。
4. 敏感信息列级加密存储。

#### 亮点怎么讲

亮点在于把医疗安全做成系统级兜底，而不是只靠 Prompt 里口头提醒模型“请谨慎”。

#### 推荐回答

> 医疗助手不能只关注回答效果，还要关注风险兜底。我做了几层控制：如果用户问题里出现高风险症状，会额外提示及时线下就医；如果检索证据不足，不强行伪装成知识库增强回答，而是明确降级为一般医学信息并给出风险提示；同时在生成后做证据校验，尽量拦截和修正不够 grounded 的内容。对于用户病史、凭证这类敏感信息，数据库侧也做了加密存储，避免直接明文落库。

#### 面试官可能会问

1. 高风险症状怎么识别？
2. 什么叫“证据不足降级回答”？
3. 证据校验怎么做？
4. 为什么医疗场景要特别强调幻觉控制？
5. 列级加密和应用层加密有什么区别？

#### 稳妥回答

1. 可以通过规则 + 风险词特征 + 节点判断组合实现，核心是高风险场景优先提醒就医。
2. 就是不伪装成强证据回答，而是明确说明当前回答更多是一般医学信息，不能替代医生诊断。
3. 对召回证据做充分性判断，并对最终回答做 grounding check。
4. 因为医疗回答的错误代价更高，不能只看“语言上像不像正确答案”。
5. 你这里更接近应用层加密后再存库，但最终效果是实现字段级别的敏感信息保护。

#### 容易被问倒的点

不要把安全兜底说成“绝对杜绝幻觉”，只能说“尽量降低风险、抑制不 grounded 的回答、在高风险场景给出更保守的行为”。

## 5. 大厂面试官高频总追问

### 5.1 为什么用 LangGraph，不直接用 LangChain Chain

> 因为这个项目不是线性单步问答，而是有状态、有分支、有工具调用、有确认流程。LangGraph 更适合把意图识别、记忆召回、检索、工具执行和澄清节点组织成状态图，也方便做 checkpoint 和中断恢复。

### 5.2 这个项目最大的工程难点是什么

推荐从下面四个里选一个主讲：

1. 多轮对话状态复杂，尤其是预约中断后再恢复。
2. 医疗场景对证据质量要求高，简单 RAG 不够。
3. 多医院接入时的凭证隔离和故障隔离。
4. 真实工具调用不能只靠模型理解，必须做状态机式控制。

### 5.3 如果让你继续优化，你会做什么

可以回答：

1. 进一步规范指标体系，统一 README、简历和 benchmark 口径。
2. 强化长期记忆的冲突更新和过期机制，减少污染。
3. 针对医疗问答补更多高质量评测集，尤其是 groundedness 和风险分层指标。
4. 对多医院 MCP 工具增加更细粒度的 SLA 和限流保护。

### 5.4 如果线上延迟太高，你怎么排查

1. 先拆阶段：意图分类、rewrite、检索、rerank、答案生成、工具调用。
2. 看是否是 LLM 调用次数过多，是否有重复推理。
3. 看检索和 rerank 的耗时占比。
4. 看多医院工具加载是否反复握手。
5. 确认缓存、checkpoint、summary 是否真正起作用。

### 5.5 这个项目里你最有 ownership 的部分是什么

建议优先答你最熟的：

1. Skill 插件化路由
2. 三层记忆
3. 检索链路优化
4. 半受控预约状态机

不要贪多，宁可主打一到两块，其他部分说参与设计或联调。
